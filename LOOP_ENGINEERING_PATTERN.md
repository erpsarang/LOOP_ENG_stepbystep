# LOOP Engineering Pattern

## 1. 실습 요약

이 실습은 작은 Node.js CSV amount CLI를 만들고 운영 가능한 Git 작업 흐름으로 성장시키는 과정을 LOOP 단위로 수행했다. 초기에는 CLI 인자, 오류 처리, JSON 출력과 테스트를 보강했고, 이후 fixture와 기대값을 분리해 `npm run verify`를 단일 품질 게이트로 만들었다. README와 HANDOFF 문서, Git 기준점, GitHub remote와 동기화 감사도 순차적으로 추가했다.

기능 실험은 `experiment/*` 브랜치에서 진행했다. `--summary`와 `--csv` 기능은 설계, 구현, 수용 기준 감사, `--no-ff` 병합, 병합 후 검증, 일반 push, 원격 동기화 감사, 안전한 브랜치 정리의 전 과정을 거쳤다. CSV 실험에서는 조합 옵션 불일치를 발견했을 때 진행을 멈추고 별도 FIX LOOP로 수정한 뒤 재감사했다. 최종적으로 검증된 기능과 문서 이력은 master와 GitHub에 보존되고 실험 브랜치는 안전하게 정리됐다.

## 2. 사용한 기본 LOOP 구조

모든 LOOP는 다음 순서를 기본 골격으로 삼는다.

1. 목표와 완료 조건을 한 문장으로 정의한다.
2. 현재 경로, 브랜치, upstream과 remote 상태를 확인한다.
3. 변경 전에 작업 트리가 clean인지 확인한다.
4. 기준 품질 게이트인 `npm run verify`를 실행한다.
5. 한 LOOP에서 검토 가능한 작은 변경만 수행한다.
6. 테스트와 필요한 직접 실행 검증을 추가하거나 수행한다.
7. `git diff`와 변경 파일 목록으로 범위, 민감정보, 불필요한 경로를 검토한다.
8. 목적이 분명한 메시지로 커밋한다.
9. 허용된 경우 일반 `git push`로 원격에 반영한다.
10. HEAD, 로컬 추적 ref와 원격 ref의 해시를 비교한다.
11. 결과와 실패·재시도·다음 판단을 LOOP 문서에 남긴다.

이 순서는 변경 자체보다 변경 전후의 증거를 중요하게 다룬다. 구현 LOOP와 감사 LOOP를 분리하면 기능 작성자가 놓친 계약을 다음 LOOP에서 독립적으로 검증할 수 있다.

## 3. 수동 LOOP와 자율 LOOP

LOOP 1~32는 사람이 각 목표, 허용 범위와 중단 조건을 제시한 supervised LOOP였다. Codex는 한 번에 하나의 작은 작업을 수행하고 결과를 사용자에게 돌려줬다. 요구사항 해석이나 조합 옵션의 의미가 달라졌을 때 사람이 다음 LOOP에서 기준을 명확히 갱신했다.

LOOP 33~36은 Codex가 병합 준비 판단, `--no-ff` 병합, 병합 후 master push, 원격 동기화와 정리 판단을 연속 수행한 autonomous LOOP였다. 각 단계는 독립적인 중단 조건을 유지했으며 정상 조건에서만 다음 단계로 넘어갔다. 자율 진행은 안전 규칙을 완화하는 것이 아니라 이미 합의된 판단 범위 안에서 반복 확인을 자동화하는 방식이다.

LOOP 37은 병합 완료 브랜치 정리 LOOP였다. master 병합, 품질 게이트, 세 master 해시를 다시 확인한 뒤 로컬은 `git branch -d`, 원격은 delete push로 정리했다.

## 4. 자율 LOOP로 넘겨도 되는 작업

다음 작업은 범위와 완료 조건이 명확하고 되돌릴 수 있거나 검증 가능한 경우 자율 진행에 적합하다.

- 작은 기능 구현과 직접 관련된 테스트 보강
- README, HANDOFF, LOOP 계획과 실행 로그 문서화
- 기존 수용 기준에 따른 회귀 테스트와 품질 게이트 실행
- `experiment/*` 브랜치 안에서의 제한된 수정과 커밋
- master 대비 diff, 커밋, merge-base를 이용한 병합 전 감사
- 충돌 없는 합의된 `--no-ff` 병합
- 병합 전후 `npm run verify`와 대표 CLI 직접 실행
- 일반 push와 HEAD·추적 ref·원격 ref 동기화 감사
- 이미 합의된 기준에 따른 브랜치 정리 권고 판단

자율 작업에도 실행 환경의 권한 승인이 필요할 수 있다. 승인은 명령 실행의 안전 경계이고, LOOP의 논리적 자율성이나 실패 여부와는 별개다.

## 5. 사람 승인이 필요한 작업

다음 상황은 기존 범위를 넘어가거나 복구 비용이 크므로 사람의 판단 또는 명시적 승인이 필요하다.

- 요구사항이나 수용 기준의 의미가 바뀌는 경우
- 공개 API, 출력 계약, 데이터 모델 등 설계 변경
- Git 병합 충돌 해결과 어느 쪽 변경을 보존할지 결정
- `npm run verify` 실패 후 기대값을 바꿀지 구현을 고칠지 결정
- force push가 필요해 보이는 모든 상황
- 비밀 키, 토큰, 개인정보나 다른 민감정보 발견
- 운영 저장소 배포, 릴리스 또는 외부 사용자 영향 작업
- 데이터, 원격 브랜치, 릴리스 자산 등 외부 상태 삭제
- 병합 여부가 불명확하거나 안전 삭제가 거부된 브랜치 정리

## 6. 브랜치 전략

- `master`는 항상 검증 가능한 안정 브랜치로 유지한다.
- 기능 실험은 master의 동기화된 기준점에서 `experiment/<topic>`으로 분리한다.
- 실험 브랜치에서 설계, 구현, FIX와 수용 기준 감사를 완료한다.
- 병합 전 `master...HEAD` diff, 커밋 목록과 merge-base를 확인한다.
- 기능 이력을 명시적으로 보존하기 위해 `--no-ff` merge commit을 사용한다.
- 병합 전 master, 병합 후 master에서 각각 `npm run verify`를 실행한다.
- master push와 세 해시 동기화 확인 후에만 브랜치 정리를 검토한다.
- 병합된 로컬 브랜치는 `git branch -d`로만 삭제한다.
- 원격 브랜치 삭제는 master 반영과 동기화를 재확인한 뒤 별도 승인 범위에서 수행한다.
- `git branch -D`는 사용하지 않는다.

## 7. 검증 전략

`npm run verify`는 모든 주요 단계의 공통 품질 게이트다. 구현 전 기준 상태, 구현 후, 병합 전 실험 브랜치, 병합 전 master, 병합 후 master, push 후 감사와 브랜치 삭제 전에 반복 실행한다.

자동 검증만으로 출력 계약을 설명하기 어려울 때는 실제 CLI를 직접 실행한다. 기본, JSON, summary, CSV와 조합 옵션처럼 사용자가 실제로 호출할 경로를 확인하고 stdout, stderr와 종료 코드를 함께 판정한다. fixture 기대값은 한곳에서 관리해 테스트와 품질 게이트의 기준이 어긋나지 않게 한다.

Git 검증은 코드 검증과 별도로 수행한다.

- 병합 전: `git diff --stat master...HEAD`, `git log master..HEAD`, `git merge-base`
- 병합 후: merge commit의 부모, `git branch --merged master`, 커밋 그래프
- push 후: `git rev-parse HEAD`, 추적 ref, `git ls-remote` 결과 비교
- 정리 전: 품질 게이트, master 동기화, 병합 목록과 로컬·원격 브랜치 존재 재확인

## 8. 실패 시 중단 조건

아래 조건에서는 후속 변경, 커밋, 병합 또는 push로 넘어가지 않는다.

- 작업 트리가 clean이 아님
- 현재 브랜치나 프로젝트 경로가 예상과 다름
- `npm run verify` 실패
- 직접 실행 결과, stdout, stderr 또는 종료 코드 불일치
- 병합 충돌 발생
- HEAD, 추적 ref와 원격 ref 해시 불일치 또는 diverge
- 의도하지 않은 대규모 변경이나 범위 밖 파일 변경
- 문서 전용 LOOP에서 소스 변경 발생
- 민감정보 또는 불필요한 로컬 경로 발견
- 안전한 브랜치 삭제 실패
- 커밋 또는 push 실패

중단은 실패를 숨기지 않는 정상적인 LOOP 결과다. 원인과 관찰값을 보고하고 다음 FIX LOOP에서 요구사항이나 구현을 바로잡는다. 실패를 무시한 채 기대값을 낮추거나 다음 단계로 진행하지 않는다.

## 9. 금지 명령

다음 명령과 행동은 표준 LOOP에서 금지한다.

- `git branch -D`
- `git push --force`
- `git push -f`
- `git push --force-with-lease`
- 검증 없는 병합 또는 push
- 충돌을 근거 없이 자동 해결
- 실패한 품질 게이트를 무시하고 진행
- 범위 확인 없이 전체 파일을 덮어쓰기
- 동기화 확인 없이 원격 브랜치나 데이터를 삭제

## 10. 재사용 가능한 자율 LOOP 프롬프트 골격

아래 템플릿의 대괄호 항목을 다음 실험에 맞게 바꿔 사용할 수 있다.

```text
현재 프로젝트: [PROJECT_PATH]
현재 안정 브랜치: [STABLE_BRANCH]
실험 브랜치: [EXPERIMENT_BRANCH]

목표:
[작고 검증 가능한 최종 결과]

현재 상태:
- 현재 브랜치와 upstream: [KNOWN_STATE]
- 작업 트리: clean 예상
- 기준 검증: [VERIFY_COMMAND]
- 원격: [REMOTE_NAME]

허용 범위:
- 수정 가능 파일: [ALLOWED_FILES_OR_AREAS]
- 허용 작업: [IMPLEMENT / TEST / DOCS / NO_FF_MERGE / NORMAL_PUSH]
- 정상 조건이면 [LOOP_RANGE]를 연속 진행

공통 절차:
1. 프로젝트 경로, 현재 브랜치와 clean 상태를 확인한다.
2. remote/upstream과 로컬·원격 해시를 확인한다.
3. 변경 전 [VERIFY_COMMAND]를 실행한다.
4. 목표를 만족하는 최소 변경만 수행한다.
5. 테스트와 대표 직접 실행 [DIRECT_COMMANDS]를 수행한다.
6. diff, 변경 파일, 민감정보와 불필요한 경로를 검토한다.
7. [COMMIT_MESSAGE]로 커밋한다.
8. 허용된 경우 일반 git push를 수행한다.
9. HEAD, 추적 ref와 원격 ref 해시 일치를 확인한다.
10. LOOP_PLAN.md와 LOOP_LOG.md에 근거와 결과를 기록한다.

중단 조건:
- 예상 경로·브랜치·clean 상태 불일치
- verify 또는 직접 실행 실패
- 병합 충돌이나 로컬/원격 diverge
- 범위 밖 변경, 민감정보 또는 대규모 리팩터링 발견
- 안전한 삭제, 커밋 또는 일반 push 실패
- 요구사항이나 설계 판단이 기존 합의 범위를 벗어남

금지:
- git branch -D
- 모든 force push 변형
- 검증 없는 병합
- 충돌 자동 해결
- 실패 무시 후 진행

커밋과 push:
- 기능, 감사와 최종 결과를 목적별 커밋으로 분리한다.
- 일반 push만 사용한다.
- 삭제는 별도 명시적 승인 범위에서만 수행한다.

최종 보고:
- 완료한 LOOP와 커밋
- verify 및 직접 실행 결과
- 병합·push·동기화 상태
- 남은 브랜치와 정리 권고
- 실제 삭제 여부
- 금지 명령 미사용 여부
```

## 11. 이번 실습에서 얻은 교훈

- 작업을 작고 검증 가능한 단위로 나누면 AI가 안정적으로 진행한다.
- 완료 조건과 중단 조건을 구체적으로 주면 잘못된 가정을 발견했을 때 정확히 멈춘다.
- `npm run verify` 같은 단일 명령이 구현, 병합, push와 정리의 공통 품질 게이트 역할을 한다.
- LOOP_PLAN과 LOOP_LOG의 문서화는 다음 LOOP가 의존할 수 있는 지속적인 컨텍스트가 된다.
- 구현 LOOP와 별도 수용 기준 감사 LOOP를 두면 조합 옵션 같은 계약 불일치를 발견하기 쉽다.
- 자율 LOOP는 판단 자율성과 실행 권한 승인이 분리된다. Codex가 다음 단계를 스스로 판단하더라도 파일·Git·네트워크 쓰기 승인은 안전 경계로 남는다.
- Codex CLI의 승인 질문은 안전장치이며 자율 LOOP 실패가 아니다.
- 정상적인 중단과 FIX LOOP는 품질을 낮추는 지연이 아니라 검증 가능한 이력을 만드는 과정이다.
